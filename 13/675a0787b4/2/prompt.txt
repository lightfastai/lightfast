  1. Inngest Observability Middleware + ALS (v3 — no upgrade needed)

  Inngest v3 already supports middleware via InngestMiddleware (already exported
  from @vendor/inngest). The v3 API:

  new InngestMiddleware({
    name: "observability",
    init: () => ({
      onFunctionRun: ({ fn, ctx }) => ({
        beforeExecution: () => {
          // Open ALS scope with inngestRunId, functionName, correlationId
        },
        afterExecution: () => {
          // Emit journal, log lifecycle
        },
      }),
    }),
  })

  This would:
  - Wrap every function run in requestStore.run — all log.* calls auto-enrich with
   inngestRunId, functionName, correlationId
  - Add automatic lifecycle logging (start/complete/error) at the middleware level
   — removing the need for manual log.info("[fn] starting")/log.info("[fn]
  complete") bookends in each function
  - Feed the journal system (same pattern as tRPC)
  - Work on the current v3.52.6 without any version change

  This is the single highest-leverage next step — it extends your ALS investment
  to the code paths that need it most.

  2. Correlation ID Propagation Across Event Chains

  Once ALS is in Inngest, the next innovation: auto-inject correlationId into
  outgoing events. When entity-worker calls step.sendEvent({ name:
  "memory/entity.embed", data: { ... } }), the middleware reads the correlationId
  from ALS and attaches it to the outgoing event data. The receiving function's
  middleware extracts it and opens its own ALS scope with the same ID.

  Result: a single correlationId traces the full chain from webhook delivery
  through entity graph updates. One BetterStack query reconstructs the entire
  pipeline execution.

  3. Inngest v4 Upgrade

  After the v3 middleware proves the pattern, upgrade to v4 for:
  - Step-level observability: onStepComplete / onStepError hooks — know which step
   failed and how long each took, without adding logs inside every step.run()
  block
  - @inngest/middleware-sentry v1.0.0: onlyCaptureFinalAttempt: true (only Sentry
  after ALL retries exhausted), withIsolationScope (no tag leaking between
  concurrent runs)
  - Per-request middleware instances (no shared state bugs)

  This is a breaking upgrade affecting 12 functions, but the surface area is
  well-contained and the v3 middleware gives you a migration bridge.
