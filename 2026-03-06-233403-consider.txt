  What This Means for Architecture

  Nothing changes structurally. This is the same 5 steps. But it reveals an
  important design principle for the Coordinator Actor:

  The Coordinator doesn't have action types. It has tools and judgment.

  The temptation is to build:
  // DON'T DO THIS
  actions: {
    reviewPR: (c, prNumber) => { ... },
    assignReviewer: (c, prNumber, reviewer) => { ... },
    createLinearIssue: (c, title, assignee) => { ... },
  }

  Instead, the Coordinator has exactly one action pattern:

  // The Coordinator's run loop
  run: workflow(async (ctx) => {
    while (!c.aborted) {
      const signals = c.state.pendingSignals;

      // LLM decides WHAT to do AND HOW to do it
      // using MCP tools as its action space
      const plan = await ctx.step("reason", () =>
        reason({
          signals,
          invariants: c.state.userInvariants,
          teamState: c.state.teamProfiles,
          history: c.state.actionHistory,
          tools: MCP_TOOLS,  // the 7 tools ARE the action space
        })
      );

      // Execute the plan (which may be "review the PR yourself"
      // or "assign to Sarah" or "do nothing yet, check back in 2h")
      for (const step of plan.steps) {
        if (step.confidence >= c.state.thresholds[step.type]) {
          await ctx.step(step.id, () => mcp.call(step.tool, step.args));
        } else {
          c.broadcast("suggestion", step);
        }
      }

      // Schedule next check or wait for new signals
      if (plan.checkBack) {
        c.schedule.after(plan.checkBack, "wake");
      }
    }
  })

  The LLM generates the plan. The plan is a sequence of MCP tool calls. The
  confidence gate decides which execute and which become suggestions. The 
  Coordinator is a general-purpose agent whose capabilities are defined entirely
   by what's connected.

  Connect Slack? It can now message people. Connect Jira? It can manage tickets.

  handler. You plug in a provider, its API becomes available via
