# Research: Proxy Call Implementation Blast Radius

**Date**: 2026-04-04
**Git Commit**: fa1b286aa7e05f9dafbcd8081e720eecf4f1b7cd
**Branch**: refactor/drop-workspace-abstraction

## Research Question

Identify every file that needs to change to implement the proxy redesign described in `thoughts/shared/research/2026-04-04-proxy-schema-blast-radius.md` — replacing `proxy.search` + `proxy.execute` (installationId + endpointId) with `proxy.search` + `proxy.call` (stable connection IDs + action strings).

## Summary

**7 files require manual edits.** 1 file is auto-regenerated by a script. 3 layers auto-propagate with zero code changes (MCP walker, vendor MCP, OpenAPI docs renderer). The platform layer, all internal consumers, all provider definitions, and all backfill workers are confirmed unchanged.

The design doc's file list was accurate with one omission: `apps/www/src/content/docs/integrate/sdk.mdx` also references proxy types in its import block.

## Files That Need Manual Changes

### Layer 1: Schema Definition

#### `packages/app-validation/src/schemas/api/proxy.ts`

**Current state** — 5 exported Zod schemas + 5 inferred types:

| Export | Lines | Shape |
|---|---|---|
| `ProxyEndpointSchema` | 5–24 | `{ endpointId, method, path, description, pathParams?, timeout? }` |
| `ProxyConnectionSchema` | 26–34 | `{ installationId, provider, status, baseUrl, endpoints[] }` |
| `ProxySearchResponseSchema` | 36–40 | `{ connections: ProxyConnectionSchema[] }` |
| `ProxyExecuteRequestSchema` | 48–66 | `{ installationId, endpointId, pathParams?, queryParams?, body? }` |
| `ProxyExecuteResponseSchema` | 68–74 | `{ status, data, headers }` |

**What changes** — Replace all 5 with new action-based schemas per the design doc:
- `ProxyActionSchema` → `{ action, params[], optional[]? }`
- `ProxySearchResponseSchema` → `{ connections: [{ id, provider, actions[] }] }` — no `installationId`, no `baseUrl`, no `status`
- `ProxyCallSchema` → `{ action, params?, connection? }` — flat params, no `pathParams`/`queryParams`/`body` split
- `ProxyCallResponseSchema` → `{ status, data, headers }` — same shape as current execute response

**Barrel export**: `packages/app-validation/src/schemas/api/index.ts:2` re-exports `./proxy` — no change needed to this file unless export names change (they do: `ProxyExecute*` → `ProxyCall*`). But since it uses `export *`, only the source file changes.

### Layer 2: Contract

#### `packages/app-api-contract/src/contract.ts:64-89`

**Current state** — Two oRPC procedures under `proxy` key:
- `proxy.search` (POST `/v1/proxy/search`) — no `.input()`, output: `ProxySearchResponseSchema`
- `proxy.execute` (POST `/v1/proxy/execute`) — input: `ProxyExecuteRequestSchema`, output: `ProxyExecuteResponseSchema`

**What changes**:
- `proxy.execute` → `proxy.call` with path `/v1/proxy/call`
- Input schema: `ProxyCallSchema`
- Output schema: `ProxyCallResponseSchema`
- `proxy.search` output schema updated to new `ProxySearchResponseSchema`
- Update `summary` and `description` strings for both

**Imports to update** (lines 2-8): `ProxyExecuteRequestSchema`, `ProxyExecuteResponseSchema` → `ProxyCallSchema`, `ProxyCallResponseSchema`

### Layer 3: Server Implementation

#### `apps/app/src/lib/proxy.ts`

**Current state**:
- `proxySearchLogic` (lines 26-77): Queries `gatewayInstallations` (orgId + active), maps each through `getProvider()`, iterates `providerDef.api.endpoints`, extracts path params from URL templates, returns `{ connections: ProxyConnection[] }`
- `proxyExecuteLogic` (lines 79-121): Verifies installation ownership, delegates to `createMemoryCaller().proxy.execute(...)` forwarding `installationId`, `endpointId`, `pathParams`, `queryParams`, `body`

**What changes**:
- `proxySearchLogic` → lean response: returns `{ connections: [{ id: installationId, provider, actions }] }` where each action is `"provider.endpointId"` with param names. No `baseUrl`, no `status`, no `installationId` exposed.
- `proxyExecuteLogic` → new `proxyCallLogic`: parse `action` string (split on first `.` → provider + endpointId), resolve connection (if provided validate it, if omitted auto-resolve from org's active installations for that provider), map flat `params` to `pathParams`/`queryParams`/`body` based on endpoint definition, then call `memory.proxy.execute(...)` with the resolved `installationId` + `endpointId` + mapped params.

**Type imports** (lines 4-8): Update from `ProxyExecuteRequest`, `ProxyExecuteResponse`, `ProxySearchResponse` to new types.

#### `apps/app/src/app/(api)/lib/orpc-router.ts:30-81`

**Current state**:
- Line 31: `proxy.search` handler → `proxySearchLogic`
- Lines 47-80: `proxy.execute` handler → `proxyExecuteLogic` with error mapping

**What changes**:
- `proxy.execute` → `proxy.call`
- Import `proxyCallLogic` instead of `proxyExecuteLogic` from `~/lib/proxy`
- Error mapping may need updating for new error cases (e.g., "provider not connected", "ambiguous connection")

## Layers That Auto-Propagate (No Code Changes)

### MCP Server (`core/mcp/src/index.ts`)

`registerContractTools(server, apiContract, lf, { prefix: "lightfast" })` at line 29 walks the contract tree. When `proxy.execute` becomes `proxy.call`, the MCP tool name automatically changes from `lightfast_proxy_execute` to `lightfast_proxy_call`. Descriptions come from `contract.route.description`. Input/output schemas come from the contract's Zod schemas.

### MCP Walker (`vendor/mcp/src/index.ts:36-119`)

Generic contract walker. Uses `isContractProcedure()` for leaf detection, derives tool names from key paths (`[prefix, ...keyPath].join(sep)`). No proxy-specific code.

### OpenAPI Docs Renderer (`apps/www/src/app/(app)/(content)/_lib/openapi.ts`)

Passes committed `openapi.json` to `createOpenAPI()`. Auto-updates when the JSON is regenerated.

## Files Confirmed Unchanged

### Platform Proxy Router

`api/platform/src/router/memory/proxy.ts` — Keeps the `installationId`-based `execute` mutation (lines 89-243). The app layer resolves `action` → `installationId` + `endpointId` before calling `memory.proxy.execute`. Also keeps `listEndpoints` query (lines 35-80). Both gated by `serviceProcedure` (service JWT).

### Internal Consumers

| File | Why unchanged |
|---|---|
| `api/app/src/router/org/connections.ts` | Calls `memory.proxy.execute` directly via `createMemoryCaller()` — bypasses public API. 5 call sites (lines 220, 319, 354, 667, 731) all use `installationId` + `endpointId` directly. |
| `packages/platform-trpc/src/caller.ts` | `createMemoryCaller()` bridge — signs service JWT, returns typed tRPC caller. No proxy-specific code. |

All `enrichInstallation` / `listResources` implementations call the `executeApi` callback with `{ endpointId, pathParams?, queryParams?, body? }` — unchanged.

### Backfill Workers

`apps/app/src/app/(api)/lib/orpc-middleware.ts`, `with-api-key-auth.ts`, `with-dual-auth.ts` — auth resolution is proxy-agnostic.
