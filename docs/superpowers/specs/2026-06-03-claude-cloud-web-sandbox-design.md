# Claude Code web sandbox ‚Äî full dev stack setup

**Date:** 2026-06-03
**Status:** Draft for review
**Surface:** Claude Code on the web (claude.ai/code) cloud sandboxes. The setup script is **shared with Codex cloud** (`.codex/environments/environment.toml`) ‚Äî this is the standardization point for the two coding environments. *Not* `@claude` GitHub Actions, *not* local CLI changes.

## Problem

We want Claude Code web sandbox sessions to be able to run the **full Lightfast dev stack** (`pnpm dev`) ‚Äî app, mcp, www, platform, the MFE aggregate, local Inngest, local QStash, and the github/linear/x emulators ‚Äî against the real managed services (PlanetScale, Upstash, Clerk, etc.).

A web sandbox is a headless Anthropic-managed Linux VM that clones the repo, runs an optional **setup script**, then hands Claude a working tree. It does **not** use `.devcontainer`; network mode and secrets are configured in the web UI per environment. Today the repo is built entirely for local human dev, so a fresh sandbox cannot stand the stack up:

- `pnpm dev` is **100% Portless-orchestrated** (`portless proxy start && turbo run dev:next ‚Ä¶`); every service resolves URLs via `portless get ‚Ä¶` and runs under `portless run ‚Ä¶`. Inngest/QStash/emulator scripts call `portless get`/`portless run` directly.
- Each app boots with `with-env = dotenv -e ./.vercel/.env.development.local`. Those env files (~40 keys for `app` alone, including the **rotating** `VERCEL_OIDC_TOKEN`) are gitignored and produced by `vercel pull`. A fresh sandbox has none of them.
- Vercel linkage (`.vercel/repo.json`, `apps/*/.vercel/project.json`) is gitignored, so the sandbox starts with no link to the Vercel projects.
- **MCP config is now harness-standardized** (done as part of this work): `.mcp.json` (Claude) and `.codex/config.toml` (Codex) run an identical set ‚Äî `exa` (hosted `https://mcp.exa.ai/mcp`), `next-devtools` (npx), `inngest` (`127.0.0.1:8288`). The old local `.mcp/*.sh` scripts (exa/lightfast/postgres) and `.env.mcp`/`.env.mcp.example` are deleted, so **no MCP secrets** are needed in the sandbox.

A Claude cloud environment already exists for this repo (local pointer `env_017JcTSzELtWsWzRmXzAKqCH` in `.claude/settings.local.json`), and **Codex already has a working cloud environment** (`.codex/environments/environment.toml`, autogenerated) whose setup script corepack-installs pnpm, writes `.vercel/repo.json` inline, runs `pnpm install --frozen-lockfile`, and `vercel pull`s env per app. This work makes the Claude environment **reproducible and robust** and **standardizes both clouds on one shared `scripts/cloud/setup.sh`** extracted from Codex's proven logic ‚Äî the repo carries the versioned setup; each cloud's UI carries only the `VERCEL_TOKEN` secret + network mode.

## Goals

- A fresh web sandbox can run the full `pnpm dev` stack after the setup script completes.
- The cloud setup is **one shared script** used by both Claude cloud and Codex cloud (single source of truth).
- Env is hydrated automatically from Vercel using a **single** `VERCEL_TOKEN` secret ‚Äî no pasting ~40 vars, resilient to `VERCEL_OIDC_TOKEN` rotation.
- MCP config is identical across Claude and Codex and needs no sandbox secrets (`exa` hosted/keyless, `next-devtools` npx, `inngest` local HTTP live once the stack is up).
- The exact web-UI configuration (setup-script path, network mode, the single `VERCEL_TOKEN` secret) is documented in-repo.

## Non-goals

- `@claude` GitHub Actions / `.github/workflows/claude.yml` (explicitly out of scope this round).
- Changes to local-CLI dev behavior. The local `pnpm dev` flow must keep working unchanged.
- Making the human-browser `https://lightfast.localhost` aggregate the primary interface in a headless VM (see Portless risk).
- Committing any secret to git.

## Decisions (locked during brainstorming)

| Decision | Choice |
|---|---|
| Cloud surface | Claude web sandboxes (primary); shared setup also serves Codex cloud |
| Capability | Full dev stack (`pnpm dev`) against real managed services |
| Cloud setup | **One shared `scripts/cloud/setup.sh`** extracted from Codex's proven script; both clouds invoke it |
| Env hydration | `vercel pull` per app (env + project settings), authed by a `VERCEL_TOKEN` secret in the environment |
| Network mode | **Trusted** (default) ‚Äî broad egress incl. package registries |

## Architecture ‚Äî split of responsibilities

| Lives in the repo (versioned) | Lives in the web UI (set once per environment) |
|---|---|

## Components

### 1. `scripts/cloud/setup.sh` (shared by Claude cloud + Codex cloud; runs once at environment creation)

Extracted from Codex's proven `.codex/environments/environment.toml`, made harness-agnostic (no hard dependency on `CODEX_WORKTREE_PATH`). `set -eu` + `set -o pipefail` where available, target < ~5 min:

1. `cd` to the repo root robustly: `${CODEX_WORKTREE_PATH:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}`.
2. `corepack enable && corepack prepare pnpm@11.1.3 --activate` (match the pinned packageManager).
3. Write `.vercel/repo.json` **inline** ‚Äî org `team_oOLHPMLVuBjXyFafgsGKEZxl` + projects `lightfast-{www,app,platform}` ‚Äî establishing repo-linked mode without `vercel link` and without committing the file.
4. `pnpm install --frozen-lockfile`.
5. For each of `apps/app`, `apps/www`, `apps/platform`: `pnpm dlx vercel@latest pull <app> --yes --non-interactive --environment=development`, guarded by a skip-if-exists check (`LIGHTFAST_FORCE_VERCEL_PULL=1` forces a re-pull). The Vercel CLI reads `VERCEL_TOKEN` from the environment for auth.
6. Optional warmups guarded by `|| true` (e.g. prime turbo / next typegen) ‚Äî only if within the time budget.

**Failure policy:** steps 1‚Äì5 are critical (fail loudly so a broken environment is obvious); step 6 is best-effort. No MCP secret / `.env.mcp` step is needed ‚Äî MCP is hosted/npx now.

**Codex side:** `.codex/environments/environment.toml` is autogenerated ("do not edit manually"), so standardizing it means **reconfiguring the Codex cloud environment's setup command to `bash scripts/cloud/setup.sh`** (operator action via Codex cloud); Codex then regenerates the file as a thin shim to the shared script.

### 2. `scripts/cloud/dev.sh` (sandbox-tuned stack start)

A thin wrapper that runs the same orchestration as root `pnpm dev` but with sandbox-appropriate Portless settings (see ¬ßPortless validation). Default behavior delegates to `pnpm dev`; environment knobs let milestone 1 swap the Portless bind strategy without rewriting the root script. The root `package.json` `dev` script is **not** modified.

### 3. Standardized MCP config (`.mcp.json` + `.codex/config.toml`) ‚Äî done

Both harnesses run an identical, cloud-native MCP set; the local `.mcp/*.sh` scripts, `.env.mcp`, and `.env.mcp.example` are deleted:

- `exa` ‚Üí `{type: http, url: https://mcp.exa.ai/mcp}` (hosted, keyless) ‚Äî works in cloud and local with no secret.
- `next-devtools` ‚Üí `npx -y next-devtools-mcp@0.3.10` ‚Äî works via npx.
- `inngest` ‚Üí `{type: http, url: http://127.0.0.1:8288/mcp}` ‚Äî live once the local Inngest dev server is up (dev-time only, both surfaces).

Removed: `lightfast` + `postgres` (Claude), `lightfast` + `playwright` + `knock` (Codex). `playwright` was dropped from the canonical set. No cloud-detection gating or soft-fail is required anymore.

### 4. Docs

- New **"Cloud sandbox"** section in `AGENTS.md` (`CLAUDE.md` symlinks to it, so both harnesses get it): the setup-script path (`bash scripts/cloud/setup.sh`), Trusted network, the single `VERCEL_TOKEN` secret, the Codex-side reconfiguration note, and how to start/validate the stack.
- This spec, committed.

## Data flow

```
Environment creation (Claude cloud UI OR Codex cloud ‚Äî both call the shared script):
  environment secret (VERCEL_TOKEN)
    ‚Üí scripts/cloud/setup.sh ‚Üí write .vercel/repo.json inline ‚Üí pnpm install ‚Üí vercel pull (per app)

Session start:
  dev.sh ‚Üí portless proxy start ‚Üí turbo dev:next + inngest + qstash + emulators

    ‚Üí services reach managed backends over Trusted egress
```

## The Portless risk and validation (milestone 1)

Portless `0.12.0` is pure Node, supports `linux`, and is tagged *"for humans and agents"* ‚Äî promising, but binding `:443` and resolving `*.lightfast.localhost` inside a headless sandbox is **unverified**. This is the single highest-risk assumption and is validated **first**, in a real web sandbox session:

1. Run `setup.sh`, then `portless proxy start`; confirm it binds.
2. Run the full stack; confirm services come up and are reachable (curl / agent-browser) at the resolved URLs.

**Fallbacks if `:443`/`.localhost` won't bind headless**, in order of preference:
1. Configure Portless to a non-privileged port if it supports one.
2. Grant the bind capability in `setup.sh` (`setcap 'cap_net_bind_service=+ep'` on the node binary, or run the proxy via a privileged step).
3. Fall back to per-service direct ports without the aggregate proxy, accepting that the human-browser `lightfast.localhost` aggregate is unavailable while individual services remain reachable.

Honest expectation: the stack **runs and is programmatically reachable**; the browser aggregate is secondary in a headless VM.

## Error handling

- Setup `vercel` steps fail loudly (a half-provisioned env should be obvious, not silently degraded).
- MCP needs no sandbox secrets, so there is nothing to fail soft (`exa` is keyless; `inngest` is simply absent until the stack is up).
- `|| true` only on genuinely optional warmups.
- No secret ever written to a tracked file; `.vercel/*` stays gitignored.

## Testing / validation

- **Local regression:** `pnpm dev` still starts unchanged; both harnesses load the standardized MCP set (`exa`/`inngest`/`next-devtools`). `pnpm check && pnpm typecheck` clean.
- **Cloud milestone 1:** a real web sandbox session runs `setup.sh` then the stack; record whether Portless binds and which fallback (if any) was needed.
- **Cloud milestone 2:** confirm a representative end-to-end path works (e.g. an app route that touches DB + Clerk) using the pulled env.
