# Claude Code + Codex web sandbox — full dev stack setup (ID-free)

**Date:** 2026-06-03
**Status:** Draft for review — core mechanism validated locally (see §Local-first validation)
**Surface:** Claude Code on the web (claude.ai/code) cloud sandboxes **and** Codex cloud. One shared setup script (`scripts/cloud/setup.sh`) is the standardization point for the two coding environments. *Not* `@claude` GitHub Actions, *not* local CLI changes.

## Problem

We want Claude Code web sandbox **and** Codex cloud sessions to be able to run the **full Lightfast dev stack** (`pnpm dev`) — app, mcp, www, platform, the MFE aggregate, local Inngest, local QStash, and the github/linear/x emulators — against the real managed services (PlanetScale, Upstash, Clerk, etc.).

A web sandbox is a headless Anthropic-managed Linux VM that clones the repo, runs an optional **setup script**, then hands the agent a working tree. It does **not** use `.devcontainer`; network mode and secrets are configured in the web UI per environment. Today the repo is built entirely for local human dev, so a fresh sandbox cannot stand the stack up:

- `pnpm dev` is **100% Portless-orchestrated** (`portless proxy start && turbo run dev:next …`); every service resolves URLs via `portless get …` and runs under `portless run …`. Inngest/QStash/emulator scripts call `portless get`/`portless run` directly.
- Each app boots with `with-env = dotenv -e ./.vercel/.env.development.local`. Those env files (~40 keys for `app` alone, including the **rotating** `VERCEL_OIDC_TOKEN`) are gitignored and produced by `vercel pull`. A fresh sandbox has none of them.
- Vercel linkage (`.vercel/repo.json`, `apps/*/.vercel/project.json`) is gitignored, so the sandbox starts with no link to the Vercel projects.
- **MCP config is harness-standardized for local harnesses** (already done): `.mcp.json` (Claude/local Claude-style config) and `.codex/config.toml` (Codex CLI/IDE/local app) run an identical set — `exa` (hosted `https://mcp.exa.ai/mcp`), `next-devtools` (npx), `inngest` (`127.0.0.1:8288`). The old local `.mcp/*.sh` scripts and `.env.mcp*` are deleted. Codex cloud setup does **not** rely on `.codex/config.toml` MCP support; official Codex MCP docs document URL MCP config for CLI and IDE/local app surfaces.

A Claude cloud environment already exists for this repo (local pointer `env_017JcTSzELtWsWzRmXzAKqCH` in `.claude/settings.local.json`), and **Codex already has a working cloud environment** (`.codex/environments/environment.toml`, autogenerated) whose setup script corepack-installs pnpm, writes `.vercel/repo.json` **inline** (org + project ids hardcoded), runs `pnpm install --frozen-lockfile`, and `vercel pull`s env per app. This work makes the Claude environment reproducible, **standardizes both clouds on one shared `scripts/cloud/setup.sh`**, and — crucially — makes that script **ID-free**: no Vercel org/project ids are committed anywhere new.

## The ID-exposure model (what's secret, what isn't)

The brainstorming decision was: keep ids out of committed setup scripts, and purge the org id from git history — while accepting one unavoidable exposure.

- **Vercel project ids are *already* committed — and must be.** `apps/{app,www,platform}/vercel.json` carry each other's project ids in `relatedProjects` (a hard requirement of Vercel Microfrontends so sibling projects resolve at build). Across those three files all three MFE project ids are present in git. They cannot be hidden without breaking MFE.
- **The `apps/mcp` project id is *not* in git.** `apps/mcp` is intentionally excluded from the MFE mesh, and its `vercel.json` contains no project-id linkage, so its id only ever lived in the gitignored `.vercel/`. Keeping it out of the committed script preserves that.
- **The org/team id** previously appeared only in `.codex/environments/environment.toml` (inline `repo.json`) and in this spec. That is the "extra" exposure we remove going forward and purge from history.
- **Severity framing:** Vercel project ids and team ids are *identifiers, not credentials* — useless without a valid `VERCEL_TOKEN` scoped to the org. `VERCEL_TOKEN` is the only real secret and correctly lives only in each cloud's UI, never in git. The history rewrite is therefore defense-in-depth, not a breach fix (see §Git history remediation).

## Goals

- A fresh web sandbox (Claude **or** Codex) can run the full `pnpm dev` stack after the setup script completes.
- The cloud setup is **one shared script** invoked by both clouds (single source of truth).
- The committed setup script is **ID-free** — all Vercel identifiers come from the environment.
- Env is hydrated automatically from Vercel using namespaced linkage vars + a `VERCEL_TOKEN` secret — no pasting ~40 vars, resilient to `VERCEL_OIDC_TOKEN` rotation.
- The org id is scrubbed from tracked files going forward and purged from git history.
- Local MCP config is identical across Claude and Codex local surfaces and needs no sandbox secrets; cloud setup does not depend on Codex cloud loading project MCP config.
- The exact web-UI configuration (setup-script path, network mode, the secrets) is documented in-repo.

## Non-goals

- `@claude` GitHub Actions / `.github/workflows/claude.yml` (out of scope this round).
- Changes to local-CLI dev behavior. The local `pnpm dev` flow must keep working unchanged.
- Making the human-browser `https://lightfast.localhost` aggregate the primary interface in a headless VM (see Portless risk).
- Removing project ids from `vercel.json` (would break MFE) or committing any secret to git.

## Decisions (locked during brainstorming + local validation)

| Decision | Choice |
|---|---|
| Cloud surface | Claude web sandboxes + Codex cloud, one shared setup |
| Capability | Full dev stack (`pnpm dev`) against real managed services |
| Cloud setup | **One shared `scripts/cloud/setup.sh`**; both clouds invoke `bash scripts/cloud/setup.sh` |
| Secrecy scope | ID-free script; scrub org id from tracked files; **purge org id from git history**; project ids stay in `vercel.json` (MFE-required) |
| Linkage source | **Discrete env secrets** — `LIGHTFAST_VERCEL_ORG_ID` + one `LIGHTFAST_VERCEL_PROJECT_ID_*` per app; the script writes `.vercel/repo.json` from them |
| Var namespacing | **`LIGHTFAST_VERCEL_*`, not the reserved `VERCEL_ORG_ID`/`VERCEL_PROJECT_ID`** — a bare `VERCEL_ORG_ID` poisons every `vercel` command (it then demands `VERCEL_PROJECT_ID`). Validated locally. |
| App set | **Four** Vercel projects: `apps/{app,www,platform,mcp}` (mcp was missing from the original Codex script) |
| Scope resolution | `vercel pull <app>` resolves org/project/team from `repo.json` — no `--scope` needed even on a multi-team account (validated) |
| Network mode | **Trusted** (default) — broad egress incl. package registries |
| Vercel linkage | Setup writes `.vercel/repo.json` from env vars — not `vercel link`, not committed to git |
| MCP config | Standardized for local harnesses across `.mcp.json` + `.codex/config.toml`: `exa` (hosted, keyless), `next-devtools`, `inngest`; Codex cloud setup does not depend on project MCP config |

## Architecture — split of responsibilities

| Lives in the repo (versioned, ID-free) | Lives in each cloud's web UI (set once per environment) |
|---|---|
| `scripts/cloud/setup.sh` — shared setup (Claude + Codex) | **Setup script** field → `bash scripts/cloud/setup.sh` |
| `scripts/cloud/dev.sh` — sandbox-tuned stack start | **Network**: Trusted |
| Standardized local `.mcp.json` + `.codex/config.toml` | **Secrets (6)**: `VERCEL_TOKEN`, `LIGHTFAST_VERCEL_ORG_ID`, `LIGHTFAST_VERCEL_PROJECT_ID_{APP,WWW,PLATFORM,MCP}` |
| AGENTS.md "Cloud sandbox" section + this spec | (all other env flows from `vercel pull`) |

## Components

### 1. `scripts/cloud/setup.sh` (shared; ID-free; runs once at environment creation)

`set -eu` + `pipefail`, harness-agnostic, target < ~5 min:

1. `cd` to repo root robustly: `${CODEX_WORKTREE_PATH:-$(git rev-parse --show-toplevel 2>/dev/null || pwd)}`.
2. A single list-driven app table `"directory|name|projectIdVar"` for all four apps — the one place to maintain when an app is added/removed (the missing-mcp bug came from a hardcoded list).
3. **Require all linkage vars by name** (`LIGHTFAST_VERCEL_ORG_ID` + the four project-id vars) — fail loudly with a named message, not a bare `set -u` abort.
4. `corepack enable && corepack prepare pnpm@11.1.3 --activate`.
5. **Build `.vercel/repo.json` from the env vars** (the values land in the JSON `id`/`orgId` fields; the *source* is env, never a committed literal). File stays gitignored. `LIGHTFAST_CLOUD_SETUP_LINK_ONLY=1` writes `repo.json` and exits — used for fast, non-destructive validation.
6. `pnpm install --frozen-lockfile`.
7. For each of the four apps: `pnpm dlx vercel@latest pull <app> --yes --non-interactive --environment=development`, skip-if-exists (`LIGHTFAST_FORCE_VERCEL_PULL=1` forces re-pull). `vercel` reads `VERCEL_TOKEN` from the env in cloud; an **empty** `VERCEL_TOKEN` is unset first so local runs fall back to the `vercel login` session. Scope is resolved from `repo.json` (no `--scope`).

**Failure policy:** steps 1–7 are critical (fail loudly). No MCP secret step is needed — MCP is hosted/npx.

**Codex side:** `.codex/environments/environment.toml` is autogenerated ("do not edit manually"). Standardizing means **reconfiguring the Codex cloud environment's setup command to `bash scripts/cloud/setup.sh`**; Codex regenerates the file as a thin shim — which also removes the inline `repo.json` (and the org id) from that committed file going forward.

### 2. `scripts/cloud/dev.sh` (sandbox-tuned stack start)

A thin wrapper that runs the same orchestration as root `pnpm dev` but with sandbox-appropriate Portless settings (see §Portless validation). Default behavior delegates to `pnpm dev`; environment knobs let milestone 1 swap the Portless bind strategy without rewriting the root script. The root `package.json` `dev` script is **not** modified.

### 3. Standardized local MCP config (`.mcp.json` + `.codex/config.toml`) — done

Local harnesses run an identical MCP set; the local `.mcp/*.sh` scripts and `.env.mcp*` are deleted:

- `exa` → `{type: http, url: https://mcp.exa.ai/mcp}` (hosted, keyless).
- `next-devtools` → `npx -y next-devtools-mcp@0.3.10`.
- `inngest` → `{type: http, url: http://127.0.0.1:8288/mcp}` — live once the local Inngest dev server is up.

No cloud-detection gating or sandbox secrets required for local surfaces. Codex cloud does not use this as a setup dependency; the official Codex MCP docs document streamable HTTP `url` config for CLI and IDE extension/local app surfaces, not Codex cloud tasks.

### 4. Git history remediation (org id only)

Only the **org id literal** needs purging — project ids stay (MFE-required) and the mcp id was never in git. Surgical rewrite:

```
git filter-repo --replace-text <(printf '<org-id>==>team_REDACTED\n')
```

This redacts the literal across all history, leaving everything else intact.

**Cost (the spicy part):** the repo uses a GitHub merge queue. A rewrite means draining/pausing the queue, force-pushing `main`, rebasing/recreating open PRs and branches, and collaborators re-cloning or hard-resetting. It is **not** a guaranteed eradication — GitHub caches old commits server-side (sometimes needing GitHub Support) and existing clones/forks keep old SHAs. Because the org id is an identifier, not a credential, this is defense-in-depth.

**Shallow-clone framing (why the tree scrub is the high-value part):** cloud sandboxes clone fresh, almost certainly shallow (the local checkout is shallow), so deep history is never fetched into the VM — the org id in history is already invisible to cloud agents. Scrubbing the *current tree* (Codex `environment.toml` → shim; this spec → placeholder) is the complete fix for what the cloud agents see. The deep-history rewrite protects only the GitHub-hosted copy, full clones, and forks. Run it as a coordinated operation, or defer it and rely on the tree scrub given the low severity.

## Data flow

```
Environment creation (Claude cloud UI OR Codex cloud — both call the shared script):
  6 environment secrets (VERCEL_TOKEN + LIGHTFAST_VERCEL_ORG_ID + 4× LIGHTFAST_VERCEL_PROJECT_ID_*)
    → scripts/cloud/setup.sh → write .vercel/repo.json from env → pnpm install → vercel pull (×4 apps)
      → apps/{app,www,platform,mcp}/.vercel/.env.development.local  (hydrated)

Session start:
  dev.sh → portless proxy start → turbo dev:next + inngest + qstash + emulators
    → each next dev reads its .vercel/.env.development.local via dotenv
    → services reach managed backends over Trusted egress
```

## The Portless risk and validation (milestone 1)

Portless `0.12.0` is pure Node, supports `linux`, and is tagged *"for humans and agents"* — promising, but binding `:443` and resolving `*.lightfast.localhost` inside a headless sandbox is **unverified**. Validated first, in a real web sandbox session:

1. Run `setup.sh`, then `portless proxy start`; confirm it binds.
2. Run the full stack; confirm services come up and are reachable (curl / agent-browser) at the resolved URLs.

**Fallbacks if `:443`/`.localhost` won't bind headless**, in order of preference:
1. Configure Portless to a non-privileged port if it supports one.
2. Grant the bind capability in `setup.sh` (`setcap 'cap_net_bind_service=+ep'`, or a privileged step).
3. Per-service direct ports without the aggregate proxy — individual services reachable, aggregate unavailable.

Honest expectation: the stack **runs and is programmatically reachable**; the browser aggregate is secondary in a headless VM.

## Error handling

- Setup `vercel`/require steps fail loudly (a half-provisioned env should be obvious).
- An empty `VERCEL_TOKEN` is unset before `vercel` runs, so a blank value never breaks local fallback to the login session.
- MCP needs no sandbox secrets for local surfaces, and cloud setup has no MCP secret step to fail soft.
- No secret ever written to a tracked file; `.vercel/*` stays gitignored.

## Local-first validation (done)

The core mechanism was proven on the local machine before any cloud config, with the linkage vars moved to `~/.zshrc`:

- `setup.sh` in `LINK_ONLY` mode generates a `.vercel/repo.json` whose project set is **byte-identical** to the working one (all four apps).
- With `LIGHTFAST_VERCEL_*` set and no reserved `VERCEL_ORG_ID` present, `vercel` resolves `lightfast/lightfast-app` from `repo.json` (exit 0) on the multi-team account with no `--scope`.
- The committed script contains **no** org/project id literals (asserted by grep).
- Two bugs caught locally that would have broken the cloud: (1) `apps/mcp` was missing from the app set; (2) reserved-var poisoning from a bare `VERCEL_ORG_ID`.

## Testing / validation

- **Local regression:** `pnpm dev` still starts unchanged; local Claude/Codex surfaces load the standardized MCP set. `pnpm check && pnpm typecheck` clean.
- **Cloud milestone 1:** a real web sandbox session runs `setup.sh` then the stack; record whether Portless binds, which fallback (if any) was needed, and the cloud's actual git clone depth/identity.

## Cloud configuration checklist (for the operator)

**Both clouds** set the same **6 environment secrets** (the project-id values come from the working `.vercel/repo.json` / Vercel dashboard; the org-id value is the `team_…` team id):

- `VERCEL_TOKEN` (scoped to the lightfast team — the only true credential)
- `LIGHTFAST_VERCEL_ORG_ID`
- `LIGHTFAST_VERCEL_PROJECT_ID_APP`
- `LIGHTFAST_VERCEL_PROJECT_ID_WWW`
- `LIGHTFAST_VERCEL_PROJECT_ID_PLATFORM`
- `LIGHTFAST_VERCEL_PROJECT_ID_MCP`

Add these in each cloud's **Secrets** section, *not* Environment variables. Codex (verified against [its docs](https://developers.openai.com/codex/cloud/environments)) decrypts secrets only for the **setup-script phase** and **removes them before the agent phase** — which exactly matches `setup.sh`'s setup-only consumption (it bakes the results into on-disk `.vercel/*` files that persist for the agent). Consequence: the agent cannot re-run `vercel pull`/`setup.sh` mid-session (no creds); re-run the environment setup to re-hydrate. Do not demote them to Environment variables to enable re-pull — that re-exposes `VERCEL_TOKEN` to the agent. Secret values live only in the cloud UI, never in `.codex/environments/environment.toml` (setup-script text only).

**Claude cloud** (environment `env_017JcTSzELtWsWzRmXzAKqCH`, or a new one):
1. **Setup script** → `bash scripts/cloud/setup.sh`
2. **Network** → Trusted
3. **Secrets** → the 6 above

**Codex cloud** (same repo):
1. Reconfigure the environment's setup command → `bash scripts/cloud/setup.sh` (Codex regenerates `.codex/environments/environment.toml` as a shim).
2. Set the same 6 environment secrets.
