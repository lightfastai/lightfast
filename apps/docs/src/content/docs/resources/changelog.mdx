## Version Guidelines

### Version Numbers

We follow Semantic Versioning (MAJOR.MINOR.PATCH):

- **MAJOR**: Breaking changes requiring migration
- **MINOR**: New features, backward compatible
- **PATCH**: Bug fixes and minor improvements

### Release Cycle

- **Patch releases**: As needed for critical fixes
- **Minor releases**: Monthly feature updates
- **Major releases**: Quarterly or as needed

### Deprecation Policy

- Features are deprecated with one minor version notice
- Deprecated features are removed in next major version
- Migration guides provided for all breaking changes

## How to Update

### Quick Update

```bash
# Pull latest changes

# Update dependencies

# Run migrations if needed

# Rebuild
```

## Breaking Changes

### Future Breaking Changes

When we introduce breaking changes:

1. **Deprecation Warning**: One version advance notice
2. **Migration Guide**: Step-by-step instructions
3. **Compatibility Mode**: Temporary backward compatibility
4. **Support Period**: 3 months for migration

### Handling Breaking Changes

```typescript
// Example: API change with compatibility
export const getMessage = query({
  args: {
    // New parameter name
    messageId: v.optional(v.id("messages")),
    // Old parameter (deprecated)
    id: v.optional(v.id("messages")),
  },
  handler: async (ctx, args) => {
    const id = args.messageId ?? args.id
    if (args.id) {
      console.warn("Parameter 'id' is deprecated, use 'messageId'")
    }
    // Implementation
  },
})
```
